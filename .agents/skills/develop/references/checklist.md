# Requirement Development Checklist

## 目标

- 需求解决的是新能力、已有能力缺口，还是行为修正
- 目标是用户价值、工程价值，还是两者都有
- 是否有一句话能概括“完成后什么不同了”

## 范围

- 这次必须包含什么
- 这次明确不包含什么
- 是否存在容易被误带进来的相邻需求

## 代码落点

- 入口从哪里开始
- 共享能力应该放在哪个包
- 终端 / Web / 其他 adapter 是否只是消费共享能力
- 是否会改变现有调用链或包边界

## 外部调研

- 业界常见范式是什么
- 开源实现通常把边界放在哪一层
- 有哪些成熟做法值得借鉴，哪些坑应该避开

## 验收

- 用户能看到什么新行为
- 哪个测试最适合证明它成立
- 是否需要补充日志、错误处理或空态行为

## 执行闭环

- 需求冻结后是否能直接进入无人工介入执行
- 是否已经明确分支名、实现落点、测试策略和验证方式
- 是否存在真正会阻塞自动提交的高风险歧义

## 风险

- 是否会破坏已有兼容性
- 是否会引入跨端行为不一致
- 是否会让 future agent 更难理解代码结构

## 何时更新 `.agents`

- 新增共享边界或扩展入口
- 新增 future agent 容易走错的落点
- 变更跨包调用链、职责归属或实现路由
